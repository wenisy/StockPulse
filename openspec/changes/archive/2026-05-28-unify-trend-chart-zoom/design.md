## Context

`DailyTrendChart` 在前一轮 `add-trend-chart-controls` change 中实现为按日/按月两个 tab。用户反馈这是错误的心智模型——他们想要的是"一个连续的视图，通过缩小看更长时间范围"，类似股票软件的 1M/3M/1Y 时间范围按钮组。

后端没有"周聚合"接口，必须前端聚合。Vercel 免费 6000 次/天，需要谨慎控制 API 调用。

## Goals / Non-Goals

**Goals:**
- 统一视图：去掉"按日/按月"两个 tab 的概念
- 三档时间范围 `[1月] [3月] [1年]`，自动决定粒度
- "3月"模式前端按周聚合 3 个月数据，零后端改动
- 隐藏金额 + Brush 缩放在合适的范围下仍可用

**Non-Goals:**
- 任意自定义时间范围（如 7 天、6 个月）
- 跨年的"3月"窗口（如 11 月-1 月跨年），首版限制为同年内
- "全部历史"视图（数据量未知，先不做）
- 按周/按月切换时支持 Brush 缩放（数据点本就少）

## Decisions

### 1. 时间范围按钮组替代 tab

**决策**：导航栏左侧改为 3 个按钮 `[1月] [3月] [1年]`，使用 `Button variant="ghost" size="sm"` + 当前激活态高亮。

**理由**：直观表达"时间跨度"，符合金融图表惯例。

### 2. "3月"模式的前端聚合策略

**决策**：拉取 3 个连续月份的 `calendarData`，每周（按"最后一个交易日"作为周节点）取该周内最后一个 `hasData=true && totalValue>0` 数据点的 `totalValue` 作为周值。

**算法**：
- 拉到 3 个月所有 `CalendarData[]`，按日期升序排列
- 用 ISO week number 分组（同一年同一周为一组）
- 每组取最后一个有效数据点 → `{ date: 'YYYY-MM-DD', label: 'M/D', totalValue, totalGain (周累计), totalGainPercent (从周首日到周末日的复利计算) }`
- 周累计 `totalGain` = 该周所有日 `totalGain` 之和
- 周涨跌幅 = `周末totalValue / (周末totalValue - 周累计totalGain) - 1`（避免精度损失）

**数据源**：3 次 `fetchCalendarData(year, month)` 并发请求（`Promise.all`）。

### 3. 3 月窗口的"滑动"语义

**决策**：导航 ← / → 移动**整个 3 月窗口**，每次移动 1 个月（不是 3 个月）。

**理由**：1 个月的步长更细腻，用户可以"逐月滑动"地观察连续走势；如果一次 3 个月就跳过了关键的边界月。

### 4. 默认范围

**决策**：默认 `1月`，加载当前月数据（与现状一致），不影响零额外 API 调用的初始体验。

### 5. Brush 仅 1 月可用

**决策**：`3月` / `1年` 模式不渲染 Brush。

**理由**：3 月模式已经是周聚合（约 13 个数据点），1 年模式 12 个月度数据点，再加 Brush 缩放价值小且界面冗余。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| `3月` 模式 3 次并发 API 击中 Vercel 限额 | 用户主动选才触发；后端有 in-memory cache（10 分钟 TTL），同月请求第二次会命中 |
| `Promise.all` 中某一次失败导致整批失败 | `Promise.allSettled` 容错，部分成功仍可显示已加载的月份 |
| 周聚合算法误用 ISO week 跨年（如 2024-12-30 是 2025 年第 1 周） | 用 `(year, isoWeek)` 复合 key 分组，跨年周自动归到对应 ISO 年 |
| 3 月跨年（如 2025 年 11 月-2026 年 1 月）边界 | 首版限制：3 月窗口不能跨年（11/12 月时窗口是 9/10/11 或 10/11/12），导航越界自动 clamp |
